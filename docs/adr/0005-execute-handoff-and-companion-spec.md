# Execute hands off via a companion spec, not a copy-paste

A finished Loupe Session has to leave something behind, or the agreed change has to be re-explained
next time. The decision is persisted as **two faces of one artifact**:

- **canonical** — the portable HTML (the diagrams and grill answers), the visual review surface.
- **companion** — a terse markdown spec (`<file>.spec.md`) generated by `loupe spec`: the decisions
  list (problem, blast radius, before → after per lens, interface delta) plus a link back to the
  canonical HTML. This is the file the implementing agent re-reads and that goes into version control.

Why markdown alongside the HTML rather than HTML alone: the agent and `git` want text it can diff and
re-read; the developer wants the visual. Each face plays to its medium; the markdown stays terse and
points at the HTML rather than duplicating it.

The artifact ends with a **Decision** section — Execute / Adjust / Cancel. Execute does not dump text
for the developer to copy; it signals the agent through the existing queuePrompt round-trip to persist
the companion spec and begin implementing. Adjust keeps the loop open; Cancel drops the change. This
keeps the spec → build handoff seamless inside the coding agent (the whole point of running in one).

Recorded because the canonical/companion split and the signal-based handoff are deliberate, hard to
reverse once specs accumulate in repos, and chosen over the obvious alternatives (HTML-only; or a
copy-paste plan).
